## 1. Definitions

**"Personal Data"** means any information relating to an identified or identifiable natural person.

**"Processing"** means any operation performed on Personal Data, including collection, storage, use, disclosure, and deletion.

**"Data Subject"** means the individual to whom Personal Data relates.

**"Sub-processor"** means any third party engaged by the Processor to process Personal Data on behalf of the Controller.

**"Data Protection Laws"** means all applicable laws relating to data protection, including GDPR, CCPA, and other relevant regulations.

---

## 2. Scope and Purpose

### 2.1 Subject Matter

This Agreement governs the processing of Personal Data by Processor on behalf of Controller in connection with the provision of the Foundry process intelligence platform services ("Services").

### 2.2 Nature and Purpose of Processing

The Processor will process Personal Data for the purpose of:
- Providing the Services as described in the main service agreement
- User authentication and authorization
- Process analytics and insights generation
- System monitoring and security
- Customer support

### 2.3 Duration

Processing will continue for the duration of the service agreement, plus any retention periods required by law or specified in this Agreement.

## 3. Categories of Data and Data Subjects

### 3.1 Categories of Data Subjects

- Controller's employees and contractors
- Controller's customers and end users
- Other individuals whose data is processed through the Services

### 3.2 Categories of Personal Data

| Category | Data Elements | Purpose |
|----------|--------------|---------|
| Identity Data | Name, email, employee ID | User identification |
| Contact Data | Email address, phone number | Communication |
| Authentication Data | Password hashes, MFA tokens | Access control |
| Usage Data | Login times, feature usage | Service delivery |
| Technical Data | IP address, browser type | Security, troubleshooting |
| Process Data | Business process information | Core service functionality |

### 3.3 Special Categories of Data

The Processor does not intentionally process special categories of personal data (e.g., health data, biometric data). If Controller uploads such data, Controller is responsible for ensuring appropriate safeguards.

## 4. Processor Obligations

### 4.1 Processing Instructions

The Processor shall:
- Process Personal Data only on documented instructions from the Controller
- Inform the Controller if any instruction infringes Data Protection Laws
- Process data only for the purposes specified in this Agreement

### 4.2 Confidentiality

The Processor shall:
- Ensure that persons authorized to process Personal Data are bound by confidentiality obligations
- Not disclose Personal Data to third parties except as permitted by this Agreement

### 4.3 Security Measures

The Processor shall implement appropriate technical and organizational measures, including:

**Technical Measures:**
- [ ] Encryption at rest (AES-256)
- [ ] Encryption in transit (TLS 1.3)
- [ ] Access control (RBAC)
- [ ] Multi-factor authentication
- [ ] Regular security testing
- [ ] Vulnerability management
- [ ] Intrusion detection
- [ ] Backup and recovery

**Organizational Measures:**
- [ ] Security policies and procedures
- [ ] Employee training
- [ ] Incident response plan
- [ ] Business continuity plan
- [ ] Regular audits
- [ ] Vendor management

### 4.4 Sub-processors

The Processor shall:
- Not engage Sub-processors without prior written authorization from Controller
- Maintain a list of approved Sub-processors (Annex B)
- Ensure Sub-processors are bound by equivalent data protection obligations
- Remain liable for Sub-processor compliance

**Current Sub-processors:**
See Annex B for the current list of Sub-processors.

### 4.5 Data Subject Rights

The Processor shall assist the Controller in responding to data subject requests for:
- Access to Personal Data
- Rectification of Personal Data
- Erasure of Personal Data
- Restriction of processing
- Data portability
- Objection to processing

**Response Timeline:** Within 10 business days of receiving Controller's request.

### 4.6 Security Incidents

In the event of a Personal Data breach, the Processor shall:
- Notify the Controller without undue delay (within 24 hours)
- Provide details of the breach including:
  - Nature of the breach
  - Categories and number of data subjects affected
  - Categories and number of records affected
  - Likely consequences
  - Measures taken to address the breach
- Assist the Controller in meeting breach notification obligations

### 4.7 Data Protection Impact Assessments

The Processor shall provide reasonable assistance to the Controller for:
- Data protection impact assessments
- Prior consultation with supervisory authorities

### 4.8 Audits

The Processor shall:
- Make available information necessary to demonstrate compliance
- Allow and contribute to audits and inspections conducted by the Controller or an auditor mandated by the Controller
- Provide Controller with copies of relevant certifications (SOC 2, ISO 27001) upon request

## Annex A: Technical and Organizational Measures

### A.1 Physical Security

- Cloud provider physical security (AWS/GCP/Azure)
- SOC 2 Type II certified data centers
- 24/7 security monitoring

### A.2 Access Control

- Role-based access control (RBAC)
- Multi-factor authentication required
- Regular access reviews
- Privileged access management
- Automatic session timeout

### A.3 Encryption

| Data State | Method | Key Management |
|------------|--------|----------------|
| At rest | AES-256 | AWS KMS / HashiCorp Vault |
| In transit | TLS 1.3 | Automated certificate management |
| Backups | AES-256 | Separate key hierarchy |

### A.4 Network Security

- Network segmentation
- Firewall protection
- Intrusion detection/prevention
- DDoS protection
- VPN for administrative access

### A.5 Application Security

- Secure development lifecycle (SDLC)
- Code review requirements
- Static application security testing (SAST)
- Dynamic application security testing (DAST)
- Dependency vulnerability scanning
- Penetration testing (annual)

### A.6 Monitoring and Logging

- Centralized log management
- Security information and event management (SIEM)
- Real-time alerting
- Log retention: 90 days online, 1 year archive

### A.7 Incident Response

- Documented incident response plan
- 24/7 on-call rotation
- Regular incident response drills
- Post-incident reviews

### A.8 Business Continuity

- High availability architecture
- Automated failover
- Regular backup testing
- Disaster recovery plan (RTO: 2 hours, RPO: 5 minutes)
